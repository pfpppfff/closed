# 5秒连接超时修复说明

## 问题描述
用户反馈：5秒超时没有生效，连接一直在进行，直到报异常。程序显示"正在连接PLC..."状态持续很长时间，没有在5秒内停止。

## 问题原因分析

### 根本原因
`_plc.Open()`是S7.Net库的同步阻塞调用，它**不响应CancellationToken**。即使我们设置了5秒的CancellationToken超时，`_plc.Open()`仍然会按照S7.Net库的默认超时时间（通常30秒或更长）来执行。

### 原有代码问题
```csharp
// 原有的错误实现
await Task.Run(() =>
{
    combinedCts.Token.ThrowIfCancellationRequested(); // 这里检查有效
    _plc.Open(); // 但这里会阻塞30秒，不响应Token
    combinedCts.Token.ThrowIfCancellationRequested(); // 永远执行不到
}, combinedCts.Token);
```

问题：
1. `_plc.Open()`是同步阻塞调用，不支持取消
2. `Task.Run`的CancellationToken参数只影响任务启动，不影响任务内部执行
3. 超时检查在`_plc.Open()`完成后才执行，失去了意义

## 修复方案

### 新的超时实现
使用`Task.WhenAny`配合`Task.Delay`来实现真正的超时控制：

```csharp
// 修复后的正确实现
var connectTask = Task.Run(() =>
{
    try
    {
        _plc.Open();
        return true;
    }
    catch (Exception ex)
    {
        // 处理连接异常
        return false;
    }
});

// 等待连接完成或超时（5秒）
var timeoutTask = Task.Delay(5000, combinedCts.Token);
var completedTask = await Task.WhenAny(connectTask, timeoutTask);

if (completedTask == connectTask)
{
    // 连接任务完成
    var result = await connectTask;
    return result;
}
else
{
    // 超时了，强制关闭连接
    try { _plc?.Close(); } catch { }
    OnCommunicationError($"连接超时: 无法在5秒内连接到 {_ipAddress}");
    return false;
}
```

### 关键改进点

#### 1. **真正的超时控制**
- 使用`Task.WhenAny`等待连接任务或超时任务
- 不依赖`_plc.Open()`的内置超时
- 5秒后立即返回，不等待底层连接完成

#### 2. **强制资源清理**
- 超时时调用`_plc?.Close()`强制关闭连接
- 防止底层连接继续占用资源
- 确保PLC对象状态正确

#### 3. **取消令牌支持**
- 支持用户主动取消（点击取消按钮）
- 支持超时自动取消
- 正确区分取消原因并给出相应提示

#### 4. **异常处理优化**
- 连接失败时立即返回false
- 超时时给出明确的超时提示
- 取消时给出取消提示

## 技术细节

### Task.WhenAny的工作原理
```csharp
var task1 = ConnectTask();     // 可能需要30秒
var task2 = Task.Delay(5000);  // 5秒后完成

// 等待任何一个任务完成
var completed = await Task.WhenAny(task1, task2);

if (completed == task2)
{
    // 超时任务先完成 = 连接超时
    return false;
}
```

### 强制关闭的必要性
- `_plc.Open()`可能在后台继续执行
- 需要调用`_plc.Close()`清理资源
- 防止后续连接操作受影响

### 异常安全性
- 所有可能抛出异常的操作都用try-catch包围
- 确保超时处理不会因异常而失败
- 保证资源正确释放

## 性能影响

### 资源使用
- 每次连接创建一个额外的Task.Delay任务
- 超时后立即释放，内存影响很小
- 网络资源得到及时清理

### 响应速度
- 连接失败时从30秒减少到5秒
- 用户体验显著改善
- 界面响应更加及时
